================================================================================
  SECTION 1 — WHAT IS THIS PROJECT? (elevator pitch)
================================================================================

A portfolio-grade autonomous research system:

  User types a research question in React UI
       ↓
  FastAPI receives it, streams progress via SSE (Server-Sent Events)
       ↓
  LangGraph runs 6 AI agents in sequence (with optional clarification loop)
       ↓
  Retriever calls MCP tools → web search (DuckDuckGo via ddgs)
       ↓
  GPT-4o synthesizes a markdown report + citations + self-review
       ↓
  BugBot monitors every step, logs issues to backend/logs/

Stack: React + Vite | FastAPI | LangGraph | LangChain | OpenAI GPT-4o |
       MCP (FastMCP) | DuckDuckGo search | Docker Compose

Ports (local dev):
  Frontend  → http://localhost:5173
  Backend   → http://localhost:8000  (API docs: /api/docs)
  MCP server→ http://localhost:8001


================================================================================
  SECTION 2 — ARCHITECTURE IN 10 LINES
================================================================================

  1. React UI (frontend/src/App.jsx) POSTs query → FastAPI /api/research
  2. main.py creates SessionTrace + opens SSE stream from stream_research()
  3. LangGraph StateGraph runs 6 nodes; state dict flows node → node
  4. Planner (GPT-4o) either asks clarification OR outputs sub-queries
  5. Retriever calls MCP batch_search OR direct search_web() fallback
  6. Evidence Validator filters sources; Summarizer writes markdown report
  7. Citation Formatter picks best sources; Self-Review scores quality
  8. main.py maps each node update → SSE events (step_start, step_output, …)
  9. BugBot.check_step() after every node; flags sent as bugbot_flag SSE
 10. On end: complete SSE + bugbot_report; trace saved to backend/logs/

================================================================================
  SECTION 6 — MCP (Model Context Protocol) EXPLAINED
================================================================================

WHAT IS MCP HERE?
  MCP is a standard way for AI agents to call external TOOLS (like web search)
  over HTTP/SSE. Our retriever does NOT search the web itself — it calls an
  MCP server that exposes search tools.

WHY SEPARATE MCP SERVER?
  - Docker: mcp-server container runs independently (docker-compose.yml:6)
  - Reusable: any agent/framework can call the same tools
  - Clean boundary: search logic isolated from LangGraph pipeline

FILE: backend/mcp_server.py
  FastMCP server named "research-tools" on port 8001

  web_search() — mcp_server.py:31
    Single query → list of {title, url, snippet}
    Delegates to tools/search.py:search_web()

  news_search() — mcp_server.py:53
    News-specific search (not used by current pipeline, available as tool)

  batch_search() — mcp_server.py:91
    Runs web_search for up to 10 queries sequentially
    Returns dict: { "query text": [results...], ... }
    Used by retriever via langchain-mcp-adapters

HOW RETRIEVER CALLS MCP — backend/agents/retriever.py:28
  1. MultiServerMCPClient connects to MCP_SERVER_URL (env var)
     Default Docker: http://mcp-server:8001/sse
     Default local:  http://localhost:8001/sse
  2. client.get_tools() → find "batch_search"
  3. batch_tool.ainvoke({queries, max_results_per_query})
  4. Parse response into SearchResult Pydantic objects

FALLBACK CHAIN — backend/agents/retriever.py:86
  1. Try MCP batch_search
  2. On exception → direct search_web() for each sub-query
  3. If MCP returns 0 total results → also try direct search_web()
  WHY: Old duckduckgo_search package returned 0 results silently; ddgs works.

SEARCH IMPLEMENTATION — backend/tools/search.py:10
  search_web(query, max_results, region)
  1. Try ddgs package (primary, works in 2025+)
  2. Fallback duckduckgo_search (deprecated but kept)
  Returns: [{title, url, snippet}, ...]

================================================================================
  SECTION 9 — SYSTEM PROMPTS CATALOG (why each agent behaves as it does)
================================================================================

  PLANNER — backend/agents/planner.py:23 (_SYSTEM_PROMPT)
  --------------------------------
  Role: Research planning specialist
  Key rules:
    - action=clarify for: jargon with multiple meanings, missing scope,
      greetings/non-research ("hi", "how are you"), uncertainty
    - action=plan only when scope is clear or user already clarified
    - 5-10 sub-queries, no overlap, background→current→implications
  Examples embedded: "vectorless rag", "hi, how are you"

  EVIDENCE VALIDATOR — backend/agents/evidence_validator.py:22
  --------------------------------
  Role: Evidence validation specialist
  Key rules:
    - Judge against RESEARCH PLAN + sub-queries, not just raw user query
    - relevance_score 0-1; is_relevant if >= 0.4
    - Strict on spam; don't reject sources that match planned sub-queries

  SUMMARIZER — backend/agents/summarizer.py:23
  --------------------------------
  Role: Expert research analyst
  Key rules:
    - Synthesize (don't concatenate snippets)
    - Markdown with ## headings
    - Compare sources, note agreements/contradictions

  CITATION FORMATTER — backend/agents/citation_formatter.py:23
  --------------------------------
  Role: Academic citation specialist
  Key rules:
    - ONLY cite provided sources — never invent URLs
    - If no sources → empty lists (also enforced in code, not just prompt)
    - APA 7th edition format

  SELF-REVIEW — backend/agents/self_review.py:22
  --------------------------------
  Role: Rigorous research quality reviewer
  Key rules:
    - quality_score 0-10; approved only if >= 6.0 AND no critical hallucination risks
    - Flag unsupported claims explicitly

  STRUCTURED OUTPUT PATTERN (all LLM agents)
  -------------------------------------------
  llm = ChatOpenAI(model="gpt-4o", temperature=X)
  llm = llm.with_structured_output(SomePydanticModel)
  result = await llm.ainvoke([SystemMessage, HumanMessage])
  WHY: OpenAI returns JSON matching schema; Pydantic validates — no free-text parsing

================================================================================
  SECTION 13 — INTERVIEW TALKING POINTS
================================================================================

  "Walk me through the architecture"
  → Section 2 (10 lines) + Section 4 diagram. Emphasize: SSE streaming,
    LangGraph state machine, MCP tool layer, structured outputs.

  "Why LangGraph instead of a single LLM call?"
  → Each agent has a focused job + schema. Planner doesn't synthesize.
    Retriever doesn't need GPT. Failures are isolated per step.
    Conditional edge enables human-in-the-loop clarification.

  "How do you prevent hallucinations?"
  → Structured output (Pydantic). Citation formatter only sees real sources.
    Code guard: skip LLM if 0 sources. Self-review flags unsupported claims.
    BugBot detects fabricated citations in real time.

  "What is MCP and why use it?"
  → Section 6. Standard tool protocol. Search isolated in own service.
    Retriever uses langchain-mcp-adapters. Swappable search backend.

  "How does streaming work?"
  → POST returns SSE. main.py event_generator yields after each graph node.
    Frontend parses event:/data: lines manually (fetch stream, not EventSource).

  "How do you handle ambiguous queries?"
  → Planner action=clarify → graph END → ClarificationPanel → user answers →
    new POST with clarification field → planner proceeds with context.

  "How do you monitor quality in production?"
  → BugBot: heuristic checks per step + session verdict logs.
    Self-review agent as LLM quality gate. Session JSON traces for debugging.

  "What would you improve next?"
  → Resume paused clarification sessions (same session_id, not new POST).
    Wire GET /api/bugbot/observations to admin UI.
    Rate-limit / cache search results. Add unit tests for BugBot heuristics.
